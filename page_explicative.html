MODULE 01

Capteurs Infrarouges (IR)

Les capteurs IR émettent de la lumière infrarouge et mesurent le niveau de réflexion. Une surface blanche réfléchit beaucoup (valeur haute), une surface noire absorbe (valeur basse). C'est comme ça que le robot "voit" la ligne.

analogRead(pin)

Retourne une valeur entre 0 et 1023. Parfait pour lire l'intensité de réflexion de façon précise. Utilisé pour le PID.

digitalRead(pin)

Retourne 0 ou 1 selon un seuil matériel. Plus simple, mais moins précis pour le PID.

Nombre de capteurs

3 capteurs : simple. 5 capteurs : idéal pour le PID. 7+ capteurs : très précis pour les courbes serrées.

Disposition

Alignés perpendiculairement à la ligne, espacés de 8 à 15 mm. Hauteur sol : 3 à 8 mm.

capteurs_ir.ino — Lecture de 5 capteurs
1 // ─── Définition des pins des capteurs (de gauche à droite) ───
commentaire Les // double slash servent à écrire des commentaires. Arduino ignore tout ce qui suit les //. C'est pour TOI et ton équipe, pas pour la machine. Ici on explique ce que font les lignes suivantes.
2 const int NB_CAPTEURS = 5;
constante const int crée un nombre entier (int) qui ne changera jamais (const). On lui donne le nom NB_CAPTEURS et la valeur 5. Chaque fois qu'on écrira NB_CAPTEURS dans le code, Arduino lira "5". C'est pratique : si demain tu passes à 7 capteurs, tu changes juste ce 5 ici, et tout le code se met à jour automatiquement.
3 const int PIN_CAPTEURS[NB_CAPTEURS] = {A0, A1, A2, A3, A4};
tableau Les crochets [ ] créent un tableau — une liste de valeurs. Ici, c'est la liste des 5 numéros de pins Arduino où sont branchés les capteurs. A0, A1, A2, A3, A4 sont les pins analogiques de l'Arduino (ceux qui peuvent lire des valeurs entre 0 et 1023, pas juste 0 ou 1). Ton capteur le plus à gauche = A0, le plus à droite = A4.
4 int valeurs[NB_CAPTEURS];
variable On crée un tableau de 5 cases vides pour stocker ce que lisent les capteurs. int signifie "entier" (integer). On n'écrit pas de valeur initiale — les cases sont à 0 par défaut. Ce tableau sera rempli à chaque tour de boucle quand on lira les capteurs.
5 int SEUIL = 500; // Valeur entre blanc et noir (à calibrer !)
seuil important Le seuil est la frontière entre "blanc" et "noir". Si un capteur lit une valeur en dessous de 500 → noir (ligne). Au dessus de 500 → blanc (sol). La valeur 500 est au milieu de la plage 0-1023. Tu devras l'ajuster en fonction de ton éclairage et de tes capteurs — c'est pour ça qu'il y a "à calibrer !" dans le commentaire.
6 void lireCapteurs() {
fonction On crée une fonction appelée lireCapteurs. Une fonction, c'est comme une recette : tu lui donnes un nom, et ensuite tu peux appeler ce nom n'importe où dans le code pour exécuter tout ce qui est entre les accolades { }. void signifie que cette fonction ne retourne pas de valeur — elle fait juste une action.
7 for (int i = 0; i < NB_CAPTEURS; i++) {
boucle for La boucle for répète une action plusieurs fois. Elle a 3 parties : 1) int i = 0 → on commence à 0 · 2) i < NB_CAPTEURS → on continue tant que i est inférieur à 5 · 3) i++ → on ajoute 1 à i après chaque tour. Résultat : le bloc s'exécute 5 fois, avec i valant 0, 1, 2, 3, puis 4. Parfait pour parcourir les 5 capteurs !
8 valeurs[i] = analogRead(PIN_CAPTEURS[i]);
lecture analogRead() lit la tension sur une pin et la convertit en nombre entre 0 et 1023. On lui passe PIN_CAPTEURS[i] — par exemple au premier tour (i=0), ce sera PIN_CAPTEURS[0] = A0. Le résultat est stocké dans valeurs[i]. Donc valeurs[0] = lecture du capteur gauche, valeurs[4] = lecture du capteur droit.
9 }
fermeture L'accolade fermante } marque la fin du bloc for. Arduino retourne au début de la boucle, incrémente i, et recommence — jusqu'à ce que la condition i < 5 soit fausse.
10 }
fermeture Fin de la fonction lireCapteurs. Tout ce qui était entre ces accolades { } s'exécutera à chaque fois qu'on appelle lireCapteurs() dans le code.
11 bool surLigne(int index) {
fonction booléenne bool (boolean) = une valeur qui est soit true (vrai) soit false (faux). Cette fonction reçoit un numéro de capteur (index) et répond : "est-ce que ce capteur est sur la ligne ?" — Oui ou Non, vrai ou faux. C'est beaucoup plus lisible dans le code que de réécrire la comparaison partout.
12 return valeurs[index] < SEUIL;
retour return envoie le résultat de la fonction. Ici, valeurs[index] < SEUIL est une comparaison qui donne true si la valeur est inférieure au seuil (= surface noire = ligne) ou false sinon (= surface blanche = sol). En une seule ligne, on répond à la question "suis-je sur la ligne ?"
13 }
fermeture Fin de la fonction surLigne. On pourra maintenant écrire if (surLigne(2)) pour savoir si le capteur du milieu est sur la ligne — bien plus lisible que if (valeurs[2] < SEUIL) !
14 void setup() {
setup arduino setup() est une fonction spéciale d'Arduino. Elle s'exécute une seule fois quand l'Arduino s'allume ou est réinitialisé. C'est là qu'on configure tout : les pins, la communication série, etc. Après, Arduino passe à loop().
15 Serial.begin(9600);
communication série Serial.begin() ouvre la communication entre l'Arduino et ton ordinateur via le câble USB. Le nombre 9600 est la vitesse de communication en bauds (bits par seconde). Tu verras les messages dans le Moniteur Série du logiciel Arduino IDE. Indispensable pour le debug !
16 for (int i = 0; i < NB_CAPTEURS; i++) {
boucle for Encore une boucle for pour parcourir les 5 capteurs. On va configurer chaque pin une par une. i va de 0 à 4.
17 pinMode(PIN_CAPTEURS[i], INPUT);
pinMode pinMode() dit à Arduino comment utiliser une pin. INPUT = en entrée (on lit une valeur depuis cette pin). L'opposé serait OUTPUT (on envoie une valeur). Les capteurs envoient des données vers l'Arduino, donc les pins des capteurs doivent être en INPUT.
18 }
fermeture Fin de la boucle for. Toutes les 5 pins de capteurs sont maintenant configurées en INPUT.
19 }
fermeture Fin de setup(). Cette configuration ne se fait qu'une seule fois au démarrage. Maintenant Arduino va entrer dans loop() et y rester pour toujours.
20 void loop() {
loop arduino loop() est l'autre fonction spéciale d'Arduino. Elle tourne en boucle infinie — Arduino la répète des milliers de fois par seconde. C'est le cœur battant du robot. Tout ce qui doit se passer en continu (lire les capteurs, calculer, commander) doit être ici.
21 lireCapteurs();
appel de fonction On appelle la fonction lireCapteurs() qu'on a définie plus haut. Les parenthèses () après le nom = "exécute cette fonction maintenant". Arduino va lire les 5 capteurs et stocker les valeurs dans le tableau valeurs[].
22 for (int i = 0; i < NB_CAPTEURS; i++) {
boucle for On parcourt les 5 capteurs pour les afficher dans le moniteur série. Utile pendant le développement pour voir en temps réel ce que lisent les capteurs.
23 Serial.print(valeurs[i]);
affichage Serial.print() envoie une valeur vers ton ordinateur, visible dans le Moniteur Série. Contrairement à Serial.println(), il n'ajoute pas de retour à la ligne — les valeurs s'affichent côte à côte.
24 Serial.print(" ");
espacement On affiche deux espaces entre chaque valeur pour rendre la lecture plus facile. Le texte entre guillemets " " est une chaîne de caractères (string). Sans ces espaces, toutes les valeurs seraient collées ensemble : "10234765..." illisible !
25 }
fermeture Fin de la boucle for d'affichage.
26 Serial.println();
retour à la ligne Serial.println() sans argument envoie juste un retour à la ligne. Après avoir affiché les 5 valeurs côte à côte, on saute une ligne pour que la prochaine lecture commence sur une nouvelle ligne — sinon tout s'accumule sur une seule ligne interminable.
27 }
fermeture loop Fin de loop(). Arduino revient immédiatement au début de loop() et recommence — lire les capteurs, afficher, lire, afficher... des milliers de fois par seconde. C'est la boucle infinie du robot !
Légende : CE QUE C'EST POURQUOI ATTENTION ASTUCE
💡 Place tes 5 capteurs en arc très léger (pas complètement droit) pour mieux capter les virages avant de les entamer.

Structure du Code Arduino

Tout code Arduino tourne autour de deux fonctions. setup() s'exécute une seule fois au démarrage. loop() tourne en boucle infinie. La vitesse de cette boucle est cruciale pour le contrôle d'un robot.

ÉlémentRôleExemple
setup()Initialisation : pins, serial, calibration — une seule foispinMode(A0, INPUT);
loop()Boucle principale : lire → calculer → agir — en continulireCapteurs(); calculerPID();
Variables globalesPartagées entre toutes les fonctionsfloat erreur, correction;
#define / constConstantes de configuration, jamais modifiées#define VITESSE_BASE 150
millis()Temps écoulé en ms depuis le démarrage, non-bloquantif (millis() - t > 100) {...}
delay()Pause BLOQUANTE — à éviter dans la boucle principaledelay(500); ← dangereux !
structure_base.ino — Architecture recommandée
1 // ═══ ROBOT SUIVEUR DE LIGNES — Structure de base ═══
commentaire titre Un grand commentaire décoratif pour marquer le début du fichier. Bonne pratique : toujours identifier son fichier avec un titre, pour s'y retrouver quand on a plusieurs fichiers ouverts.
2 // ─── 1. Constantes de configuration ─────
section Les commentaires de section organisent le code en parties logiques. Quand le code fait 200 lignes, ces marqueurs permettent de trouver rapidement ce qu'on cherche.
3 #define VITESSE_BASE 150
#define #define est une directive de préprocesseur — avant de compiler, Arduino remplace chaque occurrence de VITESSE_BASE par 150. Contrairement à const int, il ne prend pas de place en mémoire. Pratique pour les constantes qu'on ajuste souvent (vitesse, seuils...). Pas de = ni de ; ici !
4 #define VITESSE_MAX 200
#define La vitesse maximale autorisée pour les moteurs. Les moteurs acceptent des valeurs de 0 à 255 via analogWrite, mais on se limite à 200 pour garder de la marge de contrôle dans les virages.
5 #define NB_CAPTEURS 5
astuce Même si on a déjà défini NB_CAPTEURS dans l'autre fichier, on le répète ici. Dans un vrai projet, on mettrait toutes ces constantes dans un seul fichier config.h. Pour l'olympiade, un seul fichier .ino suffit.
6 // ─── 2. Variables globales ───────────────
section Les variables globales sont déclarées EN DEHORS de toutes les fonctions. Ça signifie que n'importe quelle fonction (setup, loop, lireCapteurs...) peut les lire et les modifier. C'est essentiel pour partager des données entre fonctions.
7 float Kp = 0.4;
float float = nombre à virgule flottante. Contrairement à int qui ne peut stocker que des entiers (0, 1, 2...), float peut stocker des décimales (0.4, 1.25, -0.007...). Kp est le gain proportionnel du PID — on verra ça en détail dans le module 3.
8 float Ki = 0.0;
float Ki = gain intégral. On le démarre à 0.0 (désactivé). On n'active Ki qu'après avoir réglé Kp et Kd. Commencer avec Ki non nul rend le débogage beaucoup plus difficile.
9 float Kd = 1.2;
float Kd = gain dérivé. Il est souvent plus grand que Kp car son rôle est de freiner les oscillations. Un bon point de départ : Kd ≈ 2 à 5 fois Kp. Ces valeurs (Kp=0.4, Kd=1.2) sont des exemples — tu devras les ajuster pour ton robot.
10 float erreurPrecedente = 0;
mémoire PID Le PID a besoin de se souvenir de l'erreur du tour précédent pour calculer la dérivée (vitesse de changement). On l'initialise à 0 — au premier tour, on considère qu'on était bien centré.
11 float integrale = 0;
mémoire PID L'intégrale est la somme de toutes les erreurs depuis le début. Elle s'accumule à chaque tour de boucle. Elle représente "l'erreur totale accumulée dans le temps". On l'initialise à 0 au démarrage.
12 void setup() {
setup La fonction magique d'Arduino : s'exécute une seule fois au démarrage. Pense-y comme le "réveil matin" du robot — il se prépare avant de commencer à travailler.
13 Serial.begin(9600);
communication Ouvre la connexion avec l'ordinateur. Le 9600 doit correspondre au réglage dans le Moniteur Série d'Arduino IDE (menu déroulant en bas à droite). Si les valeurs affichées sont des caractères bizarres, c'est que les vitesses ne correspondent pas.
14 initialiserMoteurs();
appel On appelle une fonction qu'on aura écrite dans le module Moteurs. Elle configure les pins des moteurs en OUTPUT et les arrête. On sépare les initialisations dans des fonctions propres pour ne pas avoir un setup() de 50 lignes illisible.
15 calibrerCapteurs();
appel Lance la procédure de calibration des capteurs (module 5). Le robot bougera pendant 2 secondes pour apprendre les valeurs blanc/noir de l'environnement. C'est la raison pour laquelle tu dois attendre que le robot ait fini de se préparer avant de le poser sur la piste !
16 delay(1000);
pause exception delay(1000) = pause de 1000 millisecondes = 1 seconde. Dans setup(), delay() est acceptable car le robot ne fait rien encore. C'est une pause pour permettre à l'opérateur de reposer le robot sur la piste après la calibration. Dans loop() en revanche, delay() est à éviter absolument !
17 }
fermeture Fin de setup(). Arduino passe maintenant à loop() et n'en reviendra jamais (sauf si on réinitialise l'Arduino).
18 void loop() {
loop La boucle infinie. Sur un Arduino Uno, cette boucle peut s'exécuter jusqu'à 10 000 fois par seconde si elle est légère. Pour notre robot, on vise 500-2000 tours/seconde — plus c'est rapide, plus le robot réagit vite.
19 lireCapteurs();
étape 1 Lire — Première étape de chaque cycle : prendre les mesures. On ne peut rien décider sans savoir où est la ligne. Cette fonction remplit le tableau valeurs[] avec les nouvelles lectures fraîches.
20 float erreur = calculerErreur();
étape 2 Analyser — On calcule l'erreur : à quel point le robot est-il décalé par rapport au centre de la ligne ? Positif = ligne à droite, négatif = ligne à gauche, zéro = parfaitement centré. La variable erreur est locale à loop() — elle est recréée à chaque tour.
21 float correction = calculerPID(erreur);
étape 3 Calculer — L'algorithme PID prend l'erreur et calcule une correction. Positive = accélérer à droite, négative = accélérer à gauche. On passe erreur entre parenthèses car la fonction en a besoin pour travailler — c'est un argument.
22 commanderMoteurs(correction);
étape 4 Agir — On applique la correction aux moteurs. Moteur gauche accélère ou ralentit, moteur droit fait le contraire, pour que le robot se recentre sur la ligne. C'est le cycle complet : Lire → Analyser → Calculer → Agir.
23 }
fin de cycle Fin de loop(). Arduino revient immédiatement à la ligne 18 et recommence. En quelques millisecondes, le robot a relu les capteurs, recalculé la correction, et renvoyé de nouvelles commandes aux moteurs. C'est cette répétition ultra-rapide qui rend le suivi de ligne possible !

MODULE 03

Régulation PID

Le PID est l'algorithme qui rend ton robot fluide et précis. Il calcule une erreur (écart de la ligne par rapport au centre), puis applique une correction aux moteurs. C'est le module le plus important à maîtriser.

Correction = Kp × erreur + Ki × Σerreur + Kd × Δerreur

Kp = gain proportionnel  ·  Ki = gain intégral  ·  Kd = gain dérivé  ·  Σ = somme  ·  Δ = variation

P — Proportionnel

Correction proportionnelle à l'erreur actuelle. Trop élevé → robot oscillant comme un pendule. Trop faible → robot lent à corriger sa trajectoire.

I — Intégral

Somme des erreurs passées. Corrige les erreurs persistantes. Commencer avec Ki=0, augmenter très doucement si le robot dérive continuellement d'un côté.

D — Dérivé

Anticipation sur la variation de l'erreur. Amortit les oscillations du P. Souvent le plus important après Kp pour les robots rapides.

Calcul de l'erreur

Avec 5 capteurs (numérotés -2,-1,0,1,2), l'erreur = position moyenne pondérée. Erreur = 0 → centré. Erreur négative → ligne à gauche. Positive → ligne à droite.

pid_complet.ino — Calcul PID avec 5 capteurs
1 // Poids de chaque capteur (gauche = négatif, droite = positif)
commentaire Ce commentaire explique la logique des poids. Le capteur le plus à gauche a le poids le plus négatif (-2), le plus à droite le plus positif (+2). C'est la base du calcul d'erreur : si la ligne est à gauche, les capteurs gauches s'activent et l'erreur devient négative.
2 const int POIDS[5] = {-2, -1, 0, 1, 2};
tableau poids Un tableau de 5 poids, un par capteur. Le capteur 0 (extrême gauche) a le poids -2, le capteur 2 (milieu) a 0, le capteur 4 (extrême droite) a +2. Plus un capteur est loin du centre, plus son poids est grand — il a plus "d'influence" sur le calcul de position de la ligne.
3 float erreurPrecedente = 0;
mémoire Le PID a besoin de comparer l'erreur actuelle avec celle du tour précédent (pour la dérivée). On stocke l'ancienne erreur dans cette variable globale qui persiste entre les appels de la fonction.
4 float integrale = 0;
mémoire L'intégrale accumule toutes les erreurs. Si le robot est constamment 0.1 trop à droite, l'intégrale grossit au fil du temps et finit par pousser assez fort pour corriger ce décalage persistant.
5 float calculerErreur() {
fonction Cette fonction retourne un float (nombre décimal) — c'est la position de la ligne. Elle ne prend pas d'argument car elle lit directement les variables globales valeursCapteurs et SEUIL.
6 int somme = 0, total = 0;
variables locales On crée deux compteurs locaux (ils n'existent que dans cette fonction). somme = poids × capteur sur ligne (pour calculer la position). total = combien de capteurs voient la ligne (pour faire une moyenne). On initialise les deux à 0.
7 for (int i = 0; i < NB_CAPTEURS; i++) {
boucle On parcourt les 5 capteurs pour calculer la position de la ligne. i va de 0 à 4.
8 bool surNoir = valeursCapteurs[i] < SEUIL;
détection Pour chaque capteur, on vérifie s'il voit du noir. surNoir sera true (1) si la valeur est sous le seuil (surface noire = ligne) ou false (0) sinon. Stocker dans une variable booléenne rend la ligne suivante plus lisible.
9 somme += POIDS[i] * (int)surNoir;
calcul pondéré += signifie "ajouter à". On ajoute le poids du capteur si celui-ci voit du noir. (int)surNoir convertit true→1 et false→0. Donc si le capteur gauche (-2) voit du noir : somme += -2×1 = -2. Si le capteur droit (+2) voit du noir : somme += 2×1 = +2.
10 total += (int)surNoir;
compteur On compte simplement combien de capteurs voient du noir (true=1, false=0). Ça servira de diviseur pour faire une moyenne. Sans ça, si 2 capteurs voient du noir vs 1 seul, l'erreur serait fausse.
11 }
fin boucle Fin de la boucle. À ce stade, somme contient la somme pondérée des capteurs actifs, et total le nombre de capteurs actifs.
12 if (total == 0) return erreurPrecedente;
cas particulier important Si aucun capteur ne voit la ligne (total = 0), on ne peut pas diviser par zéro — ce serait une erreur mathématique qui planterait le programme ! À la place, on retourne la dernière erreur connue : le robot continua de chercher la ligne dans la direction où elle a disparu.
13 return (float)somme / total;
résultat On retourne la moyenne pondérée. (float)somme convertit somme en décimal avant la division pour ne pas perdre de précision (sans ça, 3÷2 = 1 et non 1.5 en arithmétique entière). Exemple : capteur 0 et 1 actifs → somme = -2 + -1 = -3, total = 2 → erreur = -1.5 → ligne fortement à gauche.
14 }
fin fonction Fin de calculerErreur(). Elle retourne un nombre entre -2 et +2 : -2 = ligne tout à gauche, 0 = ligne au centre, +2 = ligne tout à droite.
15 float calculerPID(float erreur) {
fonction PID La fonction prend l'erreur en entrée et retourne une correction. Elle reçoit float erreur comme paramètre — la valeur calculée juste avant par calculerErreur().
16 float P = erreur;
proportionnel La partie P (Proportionnelle) est simplement l'erreur elle-même. Si l'erreur est grande, P est grand → grosse correction. Si l'erreur est petite, P est petit → petite correction. Simple et direct.
17 integrale += erreur;
intégral On ajoute l'erreur actuelle à l'intégrale. Elle grossit si le robot reste longtemps d'un côté. C'est une variable globale qui persiste entre les appels — elle "accumule" la mémoire des erreurs passées.
18 integrale = constrain(integrale, -100, 100);
anti-windup critique ! constrain(valeur, min, max) empêche la valeur de dépasser les bornes. Sans ça, l'intégrale peut grossir à l'infini (on appelle ça "windup") et rendre le robot impossible à contrôler. On la limite entre -100 et +100 — si on dépasse, l'intégrale reste bloquée à la limite.
19 float I = integrale;
intégral On copie l'intégrale dans une variable locale I pour la clarté du calcul final. Ce n'est pas obligatoire techniquement, mais rend le code plus lisible.
20 float D = erreur - erreurPrecedente;
dérivé La dérivée = vitesse de changement de l'erreur. Si l'erreur était -1 et est maintenant -0.5, D = -0.5 - (-1) = +0.5 → l'erreur diminue → bonne tendance. Si l'erreur empire vite, D sera grand et ajoutera une correction préventive pour anticiper. C'est ce qui rend le robot fluide !
21 erreurPrecedente = erreur;
mémorisation On sauvegarde l'erreur actuelle pour le prochain tour de boucle. Sans ça, on ne pourrait pas calculer D au prochain tour. Cette ligne doit être APRÈS le calcul de D — si on la met avant, D serait toujours 0 !
22 return Kp*P + Ki*I + Kd*D;
formule PID C'est LA formule du PID. On multiplie chaque composante par son gain et on additionne. En pratique avec Ki=0 : Correction = 0.4×P + 1.2×D. Si l'erreur est +1 et ne change pas (D=0) : Correction = 0.4×1 = 0.4 → petite correction. Si l'erreur change vite (D=0.5) : Correction += 1.2×0.5 = +0.6 → correction plus forte.
23 }
fin PID Fin de calculerPID(). La valeur retournée sera positive si le robot doit corriger vers la droite, négative vers la gauche. Elle sera ensuite appliquée aux moteurs dans commanderMoteurs().
24 void commanderMoteurs(float correction) {
fonction moteurs On applique la correction aux deux moteurs. Si correction > 0 (ligne à droite) : moteur gauche accélère, moteur droit ralentit → robot tourne à droite. Si correction < 0 : l'inverse.
25 int vitGauche = VITESSE_BASE + correction;
calcul vitesse La vitesse du moteur gauche = vitesse de base + correction. Si correction = +20 (ligne à droite) : moteur gauche va à 150+20 = 170 (plus vite) pour faire pivoter le robot vers la droite. La correction s'ajoute à gauche et se soustrait à droite.
26 int vitDroite = VITESSE_BASE - correction;
calcul vitesse La vitesse du moteur droit = vitesse de base - correction. Si correction = +20 : moteur droit va à 150-20 = 130 (plus lent). Le robot accélère à gauche ET ralentit à droite → il pivote vers la droite pour retrouver la ligne. Symétrique parfait !
27 vitGauche = constrain(vitGauche, 0, VITESSE_MAX);
sécurité On s'assure que la vitesse reste entre 0 et VITESSE_MAX. Sans ça, si la correction est très grande, vitGauche pourrait dépasser 255 (la limite d'analogWrite) ou devenir négatif — ce qui causerait un comportement bizarre des moteurs.
28 vitDroite = constrain(vitDroite, 0, VITESSE_MAX);
sécurité Même chose pour le moteur droit. Toujours limiter les valeurs envoyées aux moteurs — les valeurs hors plage peuvent endommager le matériel ou causer des comportements imprévisibles.
29 setMoteurs(vitGauche, vitDroite);
application On appelle la fonction setMoteurs() (définie dans le module 4) avec les deux vitesses calculées. Elle s'occupera de convertir ces valeurs en signaux PWM pour le pont en H L298N qui alimente les moteurs.
30 }
fin Fin de commanderMoteurs(). Le cycle complet est bouclé : Lire → Calculer erreur → Calculer PID → Commander moteurs. Ce cycle recommence des centaines de fois par seconde !
Légende : CE QUE C'EST POURQUOI ATTENTION ASTUCE

🎮 Méthode de réglage des gains (dans cet ordre) :

  • Mettre Ki=0 et Kd=0. Augmenter Kp jusqu'à ce que le robot suive la ligne mais oscille.
  • Augmenter Kd pour réduire les oscillations. Le robot doit devenir stable.
  • Si le robot dérive légèrement sur les lignes droites, augmenter Ki très doucement.
  • Toujours tester sur le même circuit que la compétition !
🚨 Anti-windup : Sans la ligne constrain(integrale, -100, 100), l'intégrale peut exploser et rendre le robot incontrôlable. Ne jamais l'oublier !

commentaire On identifie la section de définition des pins. Il est très important de bien documenter quel fil est branché sur quelle pin Arduino.

pin PWM ENB = Enable B = vitesse du moteur droit. Pin 10 = pin PWM ✓.

12void setMoteurs(int vitG, int vitD) {

13 if (vitG >= 0) {

direction avant digitalWrite(IN1, HIGH) met la pin IN1 à +5V. digitalWrite(IN2, LOW) met IN2 à 0V. Le L298N interprète HIGH/LOW → sens de rotation avant. Si les deux sont à HIGH ou les deux à LOW → freinage électrique. Si on inverse (IN1=LOW, IN2=HIGH) → marche arrière.
15 } else {
sinon Si vitG est négatif → marche arrière. Le else gère le cas opposé au if précédent. En Arduino, if/else fonctionne exactement comme en français : "si condition → faire A, sinon → faire B".
16 digitalWrite(IN1, LOW); digitalWrite(IN2, HIGH);
direction arrière On inverse IN1 et IN2 : IN1=LOW, IN2=HIGH → sens de rotation inversé = marche arrière. Le L298N est un pont en H : il peut inverser le courant dans le moteur pour changer son sens de rotation.
17 vitG = -vitG;

18 }
fin if/else Fin du bloc de gestion de direction. Que le moteur aille en avant ou en arrière, vitG est maintenant positif et prêt pour analogWrite.

PWM vitesse analogWrite(pin, valeur) génère un signal PWM (Pulse Width Modulation) sur la pin ENA. La valeur 0 = moteur arrêté, 255 = moteur à fond. Le constrain(vitG, 0, 255) s'assure qu'on ne dépasse jamais les limites même si le PID envoie une valeur trop grande.
20 if (vitD >= 0) {
moteur droit Exactement le même raisonnement que pour le moteur gauche, mais avec les pins IN3, IN4 et ENB.
21 digitalWrite(IN3, HIGH); digitalWrite(IN4, LOW);
direction avant Direction avant moteur droit. Attention : "avant" peut signifier une rotation physiquement opposée selon le sens de montage du moteur droit. Si ton robot tourne en rond, essaie d'inverser IN3 et IN4 pour le moteur droit.
22 } else {
sinon Si vitD est négatif → marche arrière moteur droit.
23 digitalWrite(IN3, LOW); digitalWrite(IN4, HIGH);
direction arrière Direction arrière moteur droit.
24 vitD = -vitD;
valeur absolue Même chose que pour vitG : on rend la valeur positive avant de l'envoyer à analogWrite.
25 }
fin if/else Fin du bloc direction moteur droit.

PWM vitesse On applique la vitesse au moteur droit via ENB. Avec ces deux analogWrite (lignes 19 et 26), les deux moteurs ont reçu leur vitesse et leur direction. Le robot se déplace !

Calibration des Capteurs

La calibration adapte dynamiquement le seuil blanc/noir à l'environnement. Au démarrage, le robot balaie la piste pendant 2 secondes et mémorise les valeurs min et max de chaque capteur.

calibration.ino — Calibration automatique au démarrage
1int minVals[NB_CAPTEURS]; // Minimum vu = NOIR
tableau min Un tableau pour stocker la valeur MINIMALE vue par chaque capteur pendant la calibration. Le minimum correspond à la surface la plus sombre = la ligne noire. On a un minimum différent pour chaque capteur car ils peuvent avoir des sensibilités légèrement différentes.
2int maxVals[NB_CAPTEURS]; // Maximum vu = BLANC
tableau max Tableau pour la valeur MAXIMALE = surface blanche. En connaissant le min et le max de chaque capteur, on peut calculer un seuil précis au milieu et normaliser les valeurs.
3int seuilCalib[NB_CAPTEURS]; // Seuil adaptatif par capteur
avancé Au lieu d'un seul seuil global, on a un seuil différent PAR capteur. Capteur 0 peut avoir un seuil de 450, capteur 2 de 510, etc. C'est beaucoup plus précis qu'un seuil unique pour tous.
4void calibrerCapteurs() {
fonction La procédure de calibration. Elle sera appelée dans setup() une seule fois. Elle fait tourner le robot sur la piste pendant 2 secondes pour "voir" le blanc et le noir.
5 for (int i = 0; i < NB_CAPTEURS; i++) {
initialisation On initialise les tableaux min et max avec des valeurs extrêmes pour qu'ils soient correctement mis à jour dès la première lecture.
6 minVals[i] = 1023;
initialisation On met le min à la valeur MAXIMALE possible (1023). Ainsi, la première vraie lecture sera forcément inférieure et remplacera 1023. Si on mettait 0, le minimum resterait à 0 même si les capteurs ne lisent jamais si bas.
7 maxVals[i] = 0;
initialisation On met le max à 0 (valeur MINIMALE possible). La première vraie lecture sera supérieure à 0 et remplacera cette valeur initiale. Technique classique d'initialisation pour chercher des min/max.
8 }
fin init Fin de l'initialisation. Tous les capteurs ont leur min=1023 et max=0 comme point de départ.
9 Serial.println("Calibration... Bougez le robot sur la ligne");
message On affiche un message dans le Moniteur Série pour informer l'opérateur que la calibration commence. Il doit maintenant faire passer tous les capteurs sur la ligne noire ET sur le sol blanc pendant 2 secondes.

chronomètre millis() retourne le nombre de millisecondes depuis que l'Arduino s'est allumé. On capture ce moment de départ dans debut. unsigned long est un grand entier positif — nécessaire car millis() peut retourner de grandes valeurs (après 50 jours, il dépasse la capacité d'un int !)
11 int sens = 1;
variable sens On alternera le sens de pivot (1 = droite, -1 = gauche) pour que le robot balaie la piste dans les deux directions et voit à la fois le blanc et le noir.
12 while (millis() - debut < 2000) {
boucle while while(condition) répète le bloc tant que la condition est vraie. Ici : tant que le temps écoulé (millis() - debut) est inférieur à 2000ms (2 secondes). C'est l'alternative non-bloquante à delay(2000) — le robot peut faire des choses (lire les capteurs, se déplacer) pendant ces 2 secondes.
13 setMoteurs(80 * sens, -80 * sens);
pivot lent Le robot pivote lentement (vitesse 80, bien en dessous de VITESSE_MAX). Si sens=1 : setMoteurs(80, -80) → pivote à droite. Si sens=-1 : setMoteurs(-80, 80) → pivote à gauche. Les capteurs balayent de gauche à droite et vice-versa.
14 for (int i = 0; i < NB_CAPTEURS; i++) {
lecture À chaque tour de la boucle while, on lit les 5 capteurs et on met à jour les min et max. Comme le robot pivote continuellement, chaque capteur passera au-dessus de la ligne et du sol.
15 int v = analogRead(PIN_CAPTEURS[i]);
lecture capteur On lit la valeur brute du capteur i. Cette valeur est stockée temporairement dans v (variable locale à cette boucle for) pour éviter de lire deux fois la même pin.
16 if (v < minVals[i]) minVals[i] = v;
mise à jour min Si la valeur actuelle est INFÉRIEURE au minimum enregistré → elle devient le nouveau minimum. C'est l'algorithme classique de recherche de minimum : on compare et on garde la plus petite valeur vue jusqu'ici.
17 if (v > maxVals[i]) maxVals[i] = v;
mise à jour max Même chose pour le maximum : si la valeur est plus grande que le max actuel, elle devient le nouveau max. Après 2 secondes de balayage, minVals[i] contiendra la valeur la plus basse vue (noir) et maxVals[i] la plus haute (blanc).
18 }
fin for Fin de la boucle sur les 5 capteurs. On revient à la boucle while pour vérifier si les 2 secondes sont écoulées.
19 if (millis() - debut > 1000 && sens == 1) sens = -1;
changement de sens Après 1 seconde (la moitié du temps), on inverse le sens de pivot. && signifie "ET" — les deux conditions doivent être vraies. sens == 1 empêche d'inverser plusieurs fois. Résultat : le robot pivote à droite pendant 1s, puis à gauche pendant 1s.
20 }
fin while Fin de la boucle while. Les 2 secondes sont écoulées, tous les capteurs ont vu le blanc et le noir. On a maintenant des valeurs min et max fiables dans les tableaux.
21 stopper();
arrêt On arrête les moteurs. La calibration physique est terminée. Maintenant on calcule les seuils.
22 for (int i = 0; i < NB_CAPTEURS; i++) {
calcul seuils On parcourt les 5 capteurs pour calculer le seuil de chacun et l'afficher dans le Moniteur Série.

milieu de plage Le seuil optimal = exactement au milieu entre le minimum (noir) et le maximum (blanc). Si noir = 200 et blanc = 800 → seuil = (200+800)/2 = 500. Tout ce qui est en dessous de 500 = noir, au dessus = blanc. C'est le seuil le plus robuste possible.

affichage On affiche le résultat pour chaque capteur dans le Moniteur Série. Exemple d'affichage : "Capteur 2 → seuil: 487". Si les valeurs sont très proches (ex: seuil 490-510), la calibration est excellente. Si elles sont très dispersées, les capteurs ne sont peut-être pas à la bonne hauteur.
25 }
fin Fin de la boucle de calcul. La calibration est complète ! Ces seuils seront utilisés à la place de la constante SEUIL globale pour des lectures plus précises.
26}
fin fonction Fin de calibrerCapteurs(). Après cette fonction, le robot connaît parfaitement son environnement et est prêt à suivre la ligne avec une précision maximale.

lecture Toujours lire les capteurs en premier, avant de prendre toute décision. Les états sont mis à jour en fonction des lectures fraîches.
12 switch (etatActuel) {
switch transitions switch est comme une série de if/else mais plus lisible quand il y a beaucoup de cas. Il regarde la valeur d'etatActuel et saute directement au bon bloc case. Ce premier switch gère les TRANSITIONS (changements d'état).
13 case SUIVI_LIGNE:
cas Quand etatActuel vaut SUIVI_LIGNE, Arduino exécute ce qui suit jusqu'au break. C'est la routine de transitions pour l'état de suivi normal.
14 if (estIntersection()) etatActuel = INTERSECTION;
transition En mode SUIVI_LIGNE, si on détecte une intersection → on passe à l'état INTERSECTION. La priorité est: intersection d'abord, puis perte de ligne, puis virage. C'est pour ça qu'on teste dans cet ordre.
15 else if (estPerteLigne()) etatActuel = LIGNE_PERDUE;
transition else if = "sinon si". Ce bloc ne s'exécute que si le test précédent était faux. Si ce n'est pas une intersection mais qu'on perd la ligne → état LIGNE_PERDUE.
16 else if (virageGauche() || virageDroite()) etatActuel = VIRAGE_SERRÉ;
transition || signifie "OU". Si c'est un virage gauche OU un virage droite → état VIRAGE_SERRÉ. On n'a pas besoin de distinguer gauche/droite ici car le PID gèrera la correction automatiquement.
17 break;
break break est OBLIGATOIRE après chaque case dans un switch. Sans lui, Arduino continuerait d'exécuter les cases suivants (comportement "fall-through"). Avec break, on sort du switch et on passe à la suite.
18 case VIRAGE_SERRÉ:
cas virage Transitions depuis l'état VIRAGE_SERRÉ. On vérifie si la situation a changé.
19 if (estPerteLigne()) etatActuel = LIGNE_PERDUE;
transition Si on perd la ligne pendant un virage → LIGNE_PERDUE. Le virage était trop serré ou trop rapide.
20 else if (estIntersection()) etatActuel = INTERSECTION;
transition Si on tombe sur une intersection pendant un virage (ça arrive) → INTERSECTION.
21 else if (!virageGauche() && !virageDroite()) etatActuel = SUIVI_LIGNE;
transition Si le virage est terminé (plus de capteur extrême actif) → on revient à SUIVI_LIGNE. Le ! inverse : "pas de virage gauche ET pas de virage droite" = ligne redevenue droite.
22 break;
break Fin du case VIRAGE_SERRÉ. Obligatoire pour ne pas tomber dans le case suivant.
23 case LIGNE_PERDUE:
cas perte Transitions depuis LIGNE_PERDUE. La seule façon d'en sortir : retrouver la ligne.
24 if (!estPerteLigne()) etatActuel = SUIVI_LIGNE;
transition Si la ligne n'est PLUS perdue (un capteur voit du noir) → on revient directement à SUIVI_LIGNE. Simple et efficace.
25 break;
break Fin du case.
26 case INTERSECTION:
cas intersection Après une intersection, on repart automatiquement en suivi. La gestion réelle (aller tout droit, tourner...) est dans la partie ACTIONS plus bas.
27 etatActuel = SUIVI_LIGNE;
transition auto On repasse directement en SUIVI_LIGNE. gererIntersection() aura déjà fait l'action (avancer 200ms). Cette transition instantanée signifie que le switch Actions s'exécutera d'abord en INTERSECTION, puis au tour suivant en SUIVI_LIGNE.
28 break;
break Fin du case INTERSECTION.
29 }
fin switch 1 Fin du premier switch (transitions). L'état a été mis à jour. Maintenant le deuxième switch va exécuter l'action correspondante.
30 switch (etatActuel) {
switch actions Le deuxième switch gère les ACTIONS : que fait le robot dans chaque état ? On lit à nouveau etatActuel (qui vient peut-être d'être modifié par le premier switch).
31 case SUIVI_LIGNE:
action suivi En mode suivi, on applique le PID complet.
32 case VIRAGE_SERRÉ: {
fall-through intentionnel technique avancée Ici, SUIVI_LIGNE n'a pas de break avant VIRAGE_SERRÉ → les deux états partagent la même action. Le PID gère les deux cas. L'accolade { après le case permet de déclarer des variables locales à ce bloc.
33 float err = calculerErreur();
calcul erreur On calcule l'erreur actuelle avec les capteurs frais. Variable locale err (existe seulement dans ce bloc { }).
34 commanderMoteurs(calculerPID(err));
PID imbriqué On peut imbriquer les appels de fonctions. calculerPID(err) est calculé en premier, son résultat est passé directement à commanderMoteurs(). Équivalent à : float correction = calculerPID(err); commanderMoteurs(correction); — mais en une seule ligne.
35 break;
break Fin du bloc SUIVI/VIRAGE.
36 }
fin bloc Fermeture de l'accolade du bloc case.
37 case LIGNE_PERDUE: gererPerteLigne(); break;
action perte En mode LIGNE_PERDUE, on appelle gererPerteLigne() qui fait pivoter le robot pour retrouver la ligne. On peut tout mettre sur une ligne quand c'est court et lisible.
38 case INTERSECTION: gererIntersection(); break;
action intersection En mode INTERSECTION, on appelle gererIntersection() qui fait avancer le robot pendant 200ms pour traverser.
39 case ARRET: stopper(); break;
action arrêt En mode ARRET, on appelle stopper() à chaque tour de boucle. Le robot reste immobile. Pour démarrer à nouveau, il faudrait appuyer sur le bouton Reset de l'Arduino.
40 }
fin switch 2 Fin du switch des actions. Un cycle complet est terminé : lire → mettre à jour l'état → exécuter l'action. Ce cycle recommence immédiatement dans loop().
41}
fin loop Fin de loop(). La machine à états a structuré tout le comportement du robot de façon claire et maintenable. Ajouter un nouvel état = ajouter un case dans chaque switch. Simple et élégant !

Débogage & Diagnostics

Un robot qui ne marche pas sans outil de diagnostic, c'est du travail au hasard. Serial.print(), le Serial Plotter et les LED sont tes meilleurs alliés.

debug.ino — Outils de diagnostic en temps réel
1// ─── Affichage Serial Plotter ───────────────────────────────
commentaire section Le Traceur Série (Serial Plotter) est un outil graphique d'Arduino IDE (menu Outils) qui trace des courbes en temps réel. Parfait pour visualiser comment bougent les valeurs des capteurs et l'erreur PID.
2void afficherDebugPlotter() {
fonction debug Une fonction dédiée au debug. En compétition, on la commentera ou on ne l'appellera plus. Avoir une fonction séparée permet d'activer/désactiver le debug en une seule ligne.
3 for (int i = 0; i < NB_CAPTEURS; i++) {
boucle On envoie la valeur de chaque capteur. Le Traceur Série dessine une courbe par valeur numérique séparée par des tabulations.
4 Serial.print(valeursCapteurs[i]);
envoi valeur On envoie la valeur brute du capteur i (entre 0 et 1023). Le Plotter la tracera en courbe. On peut voir en temps réel comment chaque capteur réagit quand le robot passe sur la ligne.
5 Serial.print("\t");
séparateur \t = caractère de tabulation (TAB). C'est le séparateur que le Serial Plotter utilise pour distinguer plusieurs courbes. Sans ça, toutes les valeurs seraient collées ensemble et le Plotter ne pourrait pas les séparer.
6 }
fin boucle Les 5 valeurs capteurs ont été envoyées.
7 Serial.print(erreurPrecedente * 100);
erreur PID On envoie l'erreur PID multipliée par 100. Pourquoi ×100 ? L'erreur varie entre -2 et +2 — sur le graphe avec des capteurs à 0-1023, elle serait invisible ! En la multipliant par 100, elle devient visible entre -200 et +200 et on peut la lire sur le même graphe.
8 Serial.print("\t");
séparateur Tabulation après l'erreur pour qu'elle soit reconnue comme une courbe séparée.
9 Serial.println();
fin de ligne println() sans argument envoie un retour à la ligne. C'est le signal au Serial Plotter que le "paquet" de valeurs est complet — il peut tracer les points sur le graphe et passer au suivant.
10}
fin Fin de afficherDebugPlotter(). Pour l'activer, appelle cette fonction à la fin de loop(). Pour la désactiver en compétition, commente l'appel avec //.
11// ─── Mesure du temps de boucle ──────────────────────────────
commentaire Savoir combien de temps prend chaque tour de boucle est important. Si la boucle prend 50ms, le robot ne réagit qu'à 20Hz — peut-être trop lent pour les virages rapides. Objectif : moins de 5ms par boucle (200Hz+).
12unsigned long tDebut;
chronomètre Variable globale pour stocker le temps de départ de chaque boucle. unsigned long car micros() retourne un grand nombre (microsecondes depuis le démarrage — peut dépasser 4 milliards après 70 minutes).
13void loop() {
loop instrumentée Une version de loop() avec mesure de performance. On mesure le temps au début et à la fin pour calculer la durée.
14 tDebut = micros();
top départ micros() retourne les microsecondes (µs) écoulées depuis le démarrage. 1 milliseconde = 1000 microsecondes. On capture le moment au tout début de loop() pour mesurer la durée totale d'un tour.
15 // ... code normal ...
placeholder Ici se trouve tout ton code habituel (lireCapteurs, calculerPID, commanderMoteurs, etc.). Le chronomètre tourne pendant toute cette exécution.
16 long duree = micros() - tDebut;
calcul durée On soustrait le temps de départ du temps actuel = durée d'un tour de boucle en microsecondes. Si duree = 3500 → la boucle prend 3.5ms → fréquence = 285Hz. C'est excellent pour un robot suiveur de lignes !
17 Serial.print("Duree boucle: ");
label On affiche un label pour identifier la valeur dans le Moniteur Série. Sans label, une série de nombres comme "3245 3198 3301" serait difficile à interpréter.
18 Serial.print(duree);
valeur On affiche la durée en microsecondes. Surveille cette valeur pendant que tu rajoutes du code — si elle augmente trop, il faut optimiser.
19 Serial.println(" µs");
unité On affiche l'unité "µs" (microsecondes) et on passe à la ligne. Exemple d'affichage complet : "Duree boucle: 3421 µs".
20}
fin Fin de la loop() instrumentée. En compétition, retire les appels Serial pour gagner en vitesse — la communication série prend du temps !
21#define LED_DEBUG 13
LED intégrée La pin 13 de l'Arduino Uno est connectée à la LED intégrée sur la carte (la petite LED marquée "L"). On n'a pas besoin de LED externe — pratique pour des signaux d'état simples.
22void clignote(int n) {
fonction utilitaire Une fonction qui fait clignoter la LED n fois. On peut l'utiliser pour communiquer l'état du robot sans regarder le Moniteur Série : 1 clignotement = calibration OK, 2 = erreur, 3 = perte de ligne, etc.
23 for (int i = 0; i < n; i++) {
boucle n fois On répète le clignotement n fois. Si on appelle clignote(3), la boucle s'exécute 3 fois (i = 0, 1, 2).
24 digitalWrite(LED_DEBUG, HIGH); delay(100);
allumer + pause On allume la LED (HIGH = +5V) puis on attend 100ms. Le delay(100) ici est dans une fonction de diagnostic utilisée à l'arrêt — pas dans la boucle principale. Acceptable dans ce contexte.
25 digitalWrite(LED_DEBUG, LOW); delay(100);
éteindre + pause On éteint la LED (LOW = 0V) et on attend encore 100ms. Le cycle allumé/éteint de 100ms+100ms = une fréquence de clignotement de 5Hz, bien visible à l'œil nu.
26 }
fin boucle Fin de la boucle de clignotement.
27}
fin fonction Fin de clignote(). Usage : clignote(1) dans setup() après calibration → LED clignote une fois pour dire "prêt !". clignote(3) quand le robot s'arrête → 3 clignotements = "j'ai terminé le circuit". Pas besoin d'ordinateur branché !

⚡ Augmenter la vitesse

Une fois le PID stable à vitesse lente, augmenter VITESSE_BASE progressivement et réajuster Kd. Le robot le plus rapide est souvent le gagnant.

🗺️ Cartographie du circuit

Mémoriser les intersections avec un compteur. Optimiser la trajectoire selon le circuit connu à l'avance.

🔁 PID adaptatif

Changer les gains Kp/Kd selon la courbure détectée. Moins agressif en ligne droite, plus réactif dans les virages.

🚧 Capteur obstacles

Ajouter un capteur ultrasonique HC-SR04 pour détecter et contourner les obstacles. Nouvelle dimension stratégique !

📶 Télécommande Bluetooth

Module HC-05/06 pour changer les gains PID sans recompiler, ou démarrer/arrêter à distance.

⏱️ Chronométrage

Enregistrer le temps de chaque tour avec millis(). Afficher sur un écran LCD ou envoyer via Bluetooth.

📐 Formules de référence rapide
Erreur = Σ(poids[i] × surNoir[i]) / Σ(surNoir[i])
Correction = Kp×E + Ki×Σ(E) + Kd×(E - E_prec)
Vit_gauche = Vit_base + Correction
Vit_droite = Vit_base − Correction
